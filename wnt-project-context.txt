OVERVIEW
--------
A fully self-contained, single-file HTML progressive web app (PWA) for a 3-day
golf trip called the "We Needed This Invitational" — April 9–11, 2026 in Scottsdale, AZ.
The app is a real-time scoring tracker, live leaderboard, group chat, wager platform,
and notification hub for 8 players across 3 rounds at 3 different courses.

Primary files:
  golf-trip.html  — the working source file
  index.html      — always a direct copy of golf-trip.html (served at the live domain)

The app is live and password-gated at: www.weneededthis.com
Password: amasterslovestory (SHA-256 hashed client-side, stored in localStorage)

The file is fully self-contained — one HTML file with embedded CSS, JavaScript,
and all images base64-encoded inline. No framework, no build step, no server-side
code. Only external dependency is Google Fonts.

File size: ~2.1 MB (due to base64-embedded logo and app icon images)

HANDICAP STROKE CALCULATION
----------------------------
For net scoring (Days 2 & 3), strokes are allocated hole by hole:
  - Each hole has a difficulty rating 1–18 (1 = hardest, 18 = easiest)
  - A player with handicap N receives:
      floor(N/18) strokes on every hole
      +1 additional stroke on the (N mod 18) hardest holes
  - Example: Cromer (hcp 8 on Day 2) gets 1 stroke on the 8 hardest holes
             (difficulty ratings 1 through 8), 0 strokes on holes rated 9–18
  - Stroke dots appear as small gold superscript dots on each player's score cells
    to show visually which holes they receive strokes on (Days 2 & 3 only)
  - Key functions: strokesOn(hcp, h, day), net(g, hcp, h, day), phcp(k, day)

REAL-TIME SYNC
--------------
- Remote state store: JSONbin Pro (bin ID: 69c6bfa3c3097a1dd568bcda)
- All devices poll for remote updates every 10 seconds (setInterval pollRemote, 10000)
- On each poll, the app diffs SC / STATS / wagers before and after applyRemoteState()
  so it can fire alerts only for genuinely new data arriving from other devices
- localStorage used as a fallback if JSONbin is unavailable (offline mode)
- localStorage key: 'wni-data-2026'
- Last-write-wins for concurrent edits (acceptable given 8 players, each entering
  only their own scores)
- A small green dot in the header pulses when a remote score update is received
- Presence dots show which players have been active in the last 90 seconds
- Data epoch / wipe mechanism: REQUIRED_WIPE_AT timestamp constant —
  any wager, bet, or chat message with ts < REQUIRED_WIPE_AT is silently dropped
  on load, apply, and push. Bump this constant to wipe stale data across all devices.

State object shape (getState()):
  {
    scores:     SC[day][key][holeIndex] = gross score (int or null)
    stats:      STATS[day][key][holeIndex] = { fh, gir, putts, penalty }
    chat:       { messages: [...] }
    avatars:    { [key]: base64DataUrl }
    teamNames:  { [teamKey]: string }
    bets:       [...]
    drives:     { g1: [...], g2: [...] }  — D1 shamble drive tracker
    gameWagers: { [key]: {...} }
    presence:   { [key]: timestamp }
    wipeAt:     REQUIRED_WIPE_AT
    v:          1
  }


NOTIFICATION BELL & ALERT SYSTEM
----------------------------------
A yellow SVG bell icon (Tabler Icons style) sits in the top-right of the header.
- Red numbered badge shows unread alert count
- Tapping the bell opens the Notification Log drawer showing all past alerts
- Each alert has an icon, headline, sub-text, and relative timestamp
- Alerts also fire as native iOS/Android push notifications when the app is
  installed as a PWA (via Web Notifications API, iOS 16.4+ required)

Alert checks run after every remote poll (comparing prev vs current state) and
after every local score entry. The alertedEvents Set prevents duplicate firing.

  Regular birdie              🐦  "Birdie · [name]"
  Double bogey                😬  "Double bogey · [name]"

  Triple bogey                💀  "Triple bogey · [name]"

  Hot start                   🔥  Under par through first 3 holes

WAGER / BETTING SYSTEM
-----------------------
Accessible via the Bets tab. Two sub-tabs: Bets and Ledger.

There are two categories of bets: Game Wagers and Player Bets.

GAME WAGERS (gameWagers object)
  Pre-defined group bets on match outcomes. Each requires mutual acceptance
  and majority approval (vote). Game wager keys:
    d1            — Day 1 shamble winner (Group 1 vs Group 2)
    d2_pot        — Day 2 combined pot (best net team across both matches)
    d2_m1         — Day 2 Match 1 winner
    d2_m2         — Day 2 Match 2 winner
    d2_skins_1    — Day 2 Match 1 skins
    d2_skins_2    — Day 2 Match 2 skins
    d3_m0         — Day 3 Match 1 winner
    d3_m1         — Day 3 Match 2 winner
    d3_side       — Day 3 overall side (total holes won across both matches)
  Each game wager has status: 'proposed' → 'active' → 'settled'
  Auto-settle when round results are known.
  Alert fires when a wager transitions to 'active'.

PLAYER BETS (bets array)
  P2P and house bets proposed, accepted, and settled per player.

  Bet types:
    ou          — Stroke over/under (vs computed line based on hcp/slope/rating)
    birdie-ou   — Birdie count over/under
    lowgross    — Lowest gross score in the group
    lownet      — Lowest net score in the group
    h2h         — Head-to-head net (player A beats player B)
    h2h-gross   — Head-to-head gross
    cleancard   — Player posts a clean card (no double bogey or worse)
    lastplace   — Player finishes last net
    house-prop  — House (one proposer) vs buyers; stat-based prop with odds
                  (stats: birdies, putts, fairways_hit, gir, eagles, penalties)
    pool        — Group pool (everyone contributes buy-in):
                  poolType: 'birdies' (most birdies), 'lownet' (low net),
                             'eagle_pool', 'cleancard_pool'

  Bet lifecycle: proposed → (accepted / rejected) → settled
  Settled bets track who won and amount owed.

SUGGESTED BETS
  After any completed round, the app auto-generates 8–10 suggested bets for the
  next day based on prior performance data:
    - Hot hand birdie over
    - Cold player clean card
    - H2H net (best vs worst net)
    - Last place for cold player
    - Stroke O/U for cold player
    - Low net for best net player
    - House props (birdies, putts) if stat data exists
    - Most Birdies pool
    - Low Net pool

LEDGER
  The Ledger tab shows net cash owed between all players after all bets settle.
  computeSettleUp() distills all outcomes into the minimum number of transactions
  needed to zero out all balances.

SIMULATION MODE
---------------
A hidden developer tool (not visible in production UI) that populates all 3
days with realistic randomized scores, pre-set team names, and sample wagers.
Used for testing and demos. runSim() uses a seeded PRNG for reproducibility.
clearSim() resets all state to empty and pushes clean state to JSONbin.

Sim data is automatically filtered out by the REQUIRED_WIPE_AT epoch guard —
bumping that constant clears sim data from all connected devices.

GITHUB / DEPLOYMENT WORKFLOW
------------------------------
The file is ~2.1 MB due to base64-embedded images, which GitHub renders as blank
in the web UI (it shows the file as "binary-like" when over ~1MB). This is normal —
the file is still valid and functions correctly when downloaded and opened locally
or served via the live domain.

Recommended update workflow:
  1. Claude edits golf-trip.html in the working folder
  2. Claude copies golf-trip.html to index.html (they must stay in sync)
  3. User opens GitHub Desktop → sees the diff → commits → pushes
  4. The live site at www.weneededthis.com updates automatically

Do NOT use "git add -A" for this repo — only stage golf-trip.html and index.html.
Assets/ folder (logos, etc.) is intentionally untracked; all assets are embedded.
